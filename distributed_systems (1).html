SECTION A

Distributed Architectures

How nodes are arranged and interact defines the system's scalability, resilience, and complexity.

// KEY PATTERNS
  • Client–Server — clients request services from dedicated servers
  • Peer-to-Peer — every node is both client and server
  • Cluster — tightly coupled nodes behind a load balancer
  • Microservices — decomposed services talking over APIs

↳ Real world: Google Web Servers use client-server at massive scale; BitTorrent is pure peer-to-peer.

SECTION B

Interprocess Communication

Distributed nodes share no memory. All interaction happens via network messages. Click tabs to explore 4 core IPC patterns.

// 4 PATTERNS
  • Direct Msg — raw send/recv over sockets; tight coupling
  • RPC / gRPC — call remote function, get a response back
  • Message Queue — async buffer decouples sender & receiver
  • Pub / Sub — publisher broadcasts to many subscribers

↳ Real world: Netflix uses Kafka (queue); Google uses gRPC internally; AWS SNS is Pub/Sub.

SECTION C

Synchronization & Coordination

No global clock exists in a distributed system. Safely ordering events and sharing resources requires careful algorithms.

// CONCEPTS
  • Logical Clocks — Lamport timestamps order events causally
  • Mutual Exclusion — distributed locks prevent race conditions
  • Leader Election — Raft / Paxos elect one coordinator node
  • Consensus — all nodes agree on a value despite failures

↳ Real world: Apache ZooKeeper handles coordination; etcd (Kubernetes) uses Raft consensus.

SECTION D

Fault Tolerance & Load Balancing

Partial failures are unavoidable at scale. The system must detect them and gracefully continue without interruption.

// STRATEGIES
  • Replication — copies on multiple nodes; no single point of failure
  • Heartbeating — periodic signals detect dead nodes quickly
  • Checkpointing — save state periodically for recovery
  • Load Balancing — spread requests evenly across live nodes

↳ Real world: AWS Auto Scaling replaces failed EC2 instances; Hadoop YARN reschedules tasks on failure.

SECTION E

Distributed Storage & Big Data

Petabytes of data are split into blocks spread across many nodes. Replication ensures no data loss on failure.

// SYSTEMS
  • HDFS — files split into 128 MB blocks, replicated 3×
  • MapReduce — move computation to data, not data to compute
  • NoSQL — Cassandra / DynamoDB scale horizontally
  • Object Storage — S3 / GCS for unstructured data at scale

↳ Real world: Hadoop / HDFS powers Yahoo & Facebook analytics; Google Bigtable stores the Search index.
